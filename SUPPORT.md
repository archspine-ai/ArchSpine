# Support

Use the right path so issues can be handled quickly and safely.

## Open A Public Issue When

- the CLI, MCP server, or docs behave incorrectly
- you have a reproducible bug report
- you want to request a feature or workflow improvement
- you found unclear or broken public documentation

Use the repository issue templates where possible.

## Do Not Open A Public Issue When

- the report includes credentials, secrets, tokens, or private repository content
- the problem is a potential security vulnerability
- the question depends on internal company context you do not want to disclose

For security-sensitive reports, follow [SECURITY.md](./SECURITY.md).

## Before Filing

- confirm the problem on the latest build or latest published version when possible
- include exact commands, configuration choices, and observed output
- reduce the report to the smallest reproducible example
- state whether you are using a mock provider, local model, or hosted API

## Maintainer Triage Expectations

Public support is handled on a best-effort basis. Clear reproduction steps, environment details, and expected behavior will materially improve response speed.
