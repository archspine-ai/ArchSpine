# Quick Start

Get from zero to "my agent understands my codebase" in under 5 minutes.

This tutorial uses the ArchSpine repository itself as the example. Every command shown produces the output you see here.

## Prerequisites

- **Node.js >= 20** -- ArchSpine compiles to ESM and requires a modern runtime
- **A Git repository** -- ArchSpine works inside any Git-tracked project. You do not need an existing `.spine/` directory.

## Step 1: Try (Zero-Config Preview)

Before you install anything, see what ArchSpine detects in your repository. No LLM keys, no configuration, no side effects.

```bash
npx archspine try
```

This command runs three fast checks, all completely local:

1. **Adoption check** -- detects whether `.spine/` already exists
2. **Language landscape** -- scans your repository for supported languages
3. **AST preview** -- extracts import graphs and export surfaces from sample files

**Expected output:**

```
ArchSpine Try -- Zero-Config Preview
====================================

Repository: archspine
ArchSpine adopted: yes

  ArchSpine is already active in this repository.
  .spine/index/ and .spine/view/ are populated and ready for consumption.

--- Language Landscape ---

  typescript: .ts, .tsx
  yaml: .yml, .yaml

--- AST Preview (no API keys required) ---

  typescript (XX source files, .ts, .tsx)
    Sample: src/cli/help.ts
    Imports (X total):
      ...
    Exports (X total):
      ...

--- What This Means ---

  This repo is ready. Connect your agent:
    npx --yes archspine@latest mcp setup
```

If the repository has no `.spine/` yet, the advice block at the end suggests running `npx archspine init` instead.

## Step 2: Quick Scan

Run a fast, zero-dependency scan of your codebase. This step works on **any** Git repository, even before initialization.

```bash
npx archspine scan --quick
```

The quick scan uses pure AST analysis (regex-based, no LLM) and completes in under 30 seconds for most repositories. It discovers files, classifies languages, and extracts module-level structure.

**Expected output:**

```json
{
  "mode": "quick",
  "scannedAt": "2026-05-09T12:00:00.000Z",
  "fileCount": 142,
  "languageStats": {
    ".ts": 98,
    ".tsx": 12,
    ".yml": 18,
    ".json": 14
  },
  "status": "ok"
}
```

The quick scan is a lightweight health check. You can run it as often as you like -- there is no persistent state.

## Step 3: Init (Bootstrap the Control Plane)

Initialize the `.spine/` directory. This creates the control plane structure and walks you through initial configuration.

```bash
npx archspine init
```

The interactive wizard guides you through:

1. **Languages** -- selects which languages to index (auto-detected from your repo)
2. **Agent instructions** -- optionally creates a `CLAUDE.md`, `AGENTS.md`, or `GEMINI.md` file
3. **Pre-commit hook** -- optionally installs a hook for automatic sync
4. **LLM credentials** -- optionally sets up LLM provider credentials

**What gets created:**

```
.spine/
  config.json    # Your persisted configuration
  rules/         # Architecture governance rules (you edit these)
  index/         # Semantic index (generated by sync)
  view/          # Derived views (agent briefing, diagrams, etc.)
  manifest.json  # Sync state tracking
```

After initialization, verify the structure:

```bash
ls .spine/
```

## Step 4: LLM Setup (Optional)

ArchSpine uses LLMs to generate semantic summaries during sync. This step is **optional** because you can use scan-only mode, but we recommend it for the richest agent experience.

```bash
spine llm setup
```

The interactive prompt lets you choose from eight providers: **OpenAI**, **DeepSeek**, **OpenRouter**, **Groq**, **Gemini**, **Ollama**, **LM Studio**, or a custom OpenAI-compatible endpoint.

To skip the wizard and set values directly:

```bash
spine llm --project set provider openai
spine llm --project set model gpt-4o
spine llm --project set api-key sk-...
```

Or use environment variables:

```bash
export SPINE_PROVIDER=openai
export SPINE_MODEL=gpt-4o
export SPINE_API_KEY=sk-...
```

Verify the connection:

```bash
spine llm test
```

```
  LLM connection verified (provider: openai, model: gpt-4o)
```

If you prefer a local model with no API key and no network, see the [Local LLM guide](../how-to/local-llm.md).

## Step 5: Sync (Generate the Semantic Index)

The sync command scans your source files, extracts ASTs, calls the LLM (if configured), and produces structured summaries for every tracked file.

```bash
spine sync
```

**What happens:**

1. The scanner discovers all source files in your repository
2. Each file gets AST-extracted and summarized (by LLM if configured, otherwise structural-only)
3. Summaries are written to `.spine/index/` as structured JSON
4. Views are derived: agent briefing, architecture diagram, risk hotspots, and more

**Expected output (truncated):**

```
Syncing .spine/ control plane for archspine...

Scanning source files... found XXX files
Resolving delta against manifest...
Files to process: XX (new), XX (changed), XX (unchanged)

[=====>              ] 25% -- src/cli/index.ts done
[==========>         ] 50% -- src/engines/check.ts done
[===============>    ] 75% -- src/services/sync-service.ts done
[====================] 100%

Sync complete.
  * XX new/changed files summarized
  * XX files unchanged (skipped)
  * Views derived: public-surface, risk-hotspots, architecture-diagram,
    project-health, agent-briefing, change-impact
```

The first sync may take a few minutes if an LLM is configured. Subsequent syncs are **incremental** and only process changed files.

## Step 6: MCP (Connect Your Agent)

This is the final step and the most important one. ArchSpine is **MCP-first** -- the control plane is designed to be consumed by AI agents through the Model Context Protocol.

```bash
spine mcp setup
```

This command detects your environment and writes the correct MCP configuration to make the `archspine` server available to your agent.

**Expected output:**

```
ArchSpine MCP Setup
===================

Resolved command: npx --yes archspine@latest

Writing configurations...
  Claude Code (project): .mcp.json
  Claude Code (global): ~/.claude/mcp.json

Next steps:
  1. Restart your IDE or Claude Desktop for changes to take effect
  2. Verify the connection: look for "archspine" in the MCP server list
  3. Try an MCP tool: ask your agent "what modules are in this project?"
```

After restarting your agent session, the agent has access to **21 MCP tools**, **4 resource URIs**, and **2 prompts** -- all reading from the control plane with no additional LLM calls at runtime. No more feeding entire source trees into prompts.

For a deeper walkthrough of MCP configuration, see [Connect MCP to Your Agent](../how-to/mcp-connect.md).

## What You Have Now

| Artifact             | Location                           | Purpose                                                                                                          |
| -------------------- | ---------------------------------- | ---------------------------------------------------------------------------------------------------------------- |
| Control plane config | `.spine/config.json`               | Your settings (LLM provider, enabled views)                                                                      |
| Architecture rules   | `.spine/rules/*.yml`               | Human-authored governance rules that agents can query                                                            |
| Semantic index       | `.spine/index/*.json`              | Per-file structured summaries (one JSON file per source file)                                                    |
| Derived views        | `.spine/view/data/*.json`          | Pre-computed: public surface, risk hotspots, architecture diagram, project health, agent briefing, change impact |
| MCP server           | 21 tools + 4 resources + 2 prompts | STDIO server, fully read-only, zero LLM latency at runtime                                                       |
