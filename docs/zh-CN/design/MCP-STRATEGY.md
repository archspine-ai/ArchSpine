# ArchSpine MCP 策略

本文档描述 MCP 既作为产品交付层，也作为潜在商业化路径时的定位。

## 本地 MCP 与云端 MCP

部署形态决定了 MCP 是单纯的 adoption 基础设施，还是可收费的产品层。

| 维度     | 本地 MCP             | 云端 MCP                 |
| -------- | -------------------- | ------------------------ |
| 传输方式 | STDIO                | HTTPS / SSE / WebSocket  |
| 部署位置 | 开发者本机           | 托管控制面               |
| 作用范围 | 单个本地仓库         | 多仓库、跨服务、组织级   |
| 成本模型 | 用户自付模型调用成本 | 可打包数据和模型访问能力 |
| 战略角色 | 促进采用、建立标准   | 支撑企业协作与变现       |

## 阶段一：本地 MCP

开源 CLI 应优先让本地 MCP 足够直接、低摩擦。

价值：

- 不需要自建服务器
- 首次体验强
- 接入路径简单，更容易形成采用

这也是当前 `v1.0.x` 线路最合理的选择。

## 阶段二：托管 MCP

当团队开始需要跨仓库推理时，本地 MCP 就不够了，这时托管控制面才有意义。

潜在价值：

- 组织级 `.spine` 聚合
- 跨服务依赖查询
- 集中治理与审计历史
- 面向企业 Agent 工具的鉴权远程访问

## 战略结论

正确节奏应该是：

1. 用本地 MCP 建立工作流和采用面
2. 证明 `.spine` 作为仓库契约的价值
3. 只有在组织级场景真实成立后，再把 MCP 扩展成云端控制面
