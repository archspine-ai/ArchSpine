# ArchSpine 项目概览与开发者指南

## 文档目的

本文档是 ArchSpine 项目的核心 README 和开发者入门指南，阐述项目存在的原因、功能以及如何有效参与贡献。它是 **仓库结构、构建流程、代码规范与治理规则** 的单一权威来源，供人类开发者和 AI 智能体共同遵循。

## 目标读者

- **主要读者：** 直接参与 ArchSpine 工具链开发的开发者，以及需要理解仓库结构、构建命令和贡献协议的 AI 智能体。
- **次要读者：** 探索代码库的贡献者、审阅者，以及任何需要掌握 `.spine/` 背后架构理念的人。

## 本文档锚定的关键决策与工作流

### 1. ArchSpine 是什么（以及不是什么）

ArchSpine 是一个 **开源协议与工具链**，通过在 Git 仓库内构建物理化的 `.spine/` 控制面，使代码库对 AI 辅助工程具备可查询、可治理、可审计的能力。本文档不是用户手册或详细 API 参考——这些内容位于 `docs/` 子目录。

### 2. 技术栈与构建流程

项目使用 **TypeScript 5（严格模式，ESM）** 运行于 **Node.js ≥20**，测试采用 **Vitest**，代码质量由 **ESLint + Prettier** 保证，文档基于 **VitePress**。构建、测试、检查、验证、同步等关键命令均在 README 中标准化。

### 3. 代码规范与架构分层

- 使用 ESM 模块，TypeScript 源码显式添加 `.js` 扩展名，`strict: true`，缩进 2 空格。
- 变量/函数使用 `camelCase` 命名，类/类型使用 `PascalCase` 命名。
- 提交信息遵循 Conventional Commits 规范，允许可选的作用域（如 `feat(cli): 添加交互式向导`）。
- 架构分层明确：`cli/`（入口与命令分发，薄层）、`core/`（领域类型、错误、管道、任务状态）、`engines/`（业务逻辑：扫描、规则、检查、修复、上下文）、`services/`（运行时编排：同步、检查、修复、查看）、`infra/`（基础设施：配置、数据库、LLM、MCP、I/O、提示）、`tasks/`（具体任务实现）、`types/`（协议类型定义）、`utils/`（共享工具）、`ast/`（AST 提取与语言发现）。

### 4. Git 工作流与提交纪律

- 功能分支从 `main` 分出，PR 模板必须包含摘要、动机、测试清单、文档影响。
- **关键规则：** 源码改动与 `.spine/` 同步必须分两次提交，严禁混入同一次提交。
- 生成的文件（`.spine/index/`、`.spine/atlas/`、`.spine/view/`）禁止直接编辑，应使用 `spine sync` 刷新。
- 人工审查的控制面文件（`.spine/config.json`、`rules/`）可随源码修改一并提交。

### 5. 双语文档要求

所有文档须维护 **英文与简体中文** 两个版本，入口保持一致。设计/规划类文档（位于 `docs/design/`、`docs/planning/`、`docs/archive/`）不得纳入公开导航。

### 6. AI 智能体操作规范

- 必须使用 `node dist/cli/index.js sync` 刷新 `.spine/`，禁止直接编辑生成文件。
- 提交前务必执行 `git status` 确认暂存文件分类正确。
- 注意 `spine sync` 可能耗时较长（扫描全仓库并调用 LLM），建议在后台运行或提示用户手动执行。
