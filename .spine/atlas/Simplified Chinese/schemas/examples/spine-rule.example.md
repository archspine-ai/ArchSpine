# 架构规则：禁止服务层直接导入数据库层

## 文档目的
本文档定义了 ArchSpine 项目中的一项核心架构规则：应用程序服务层和中间层不得直接导入 `src/db/` 目录下的数据库适配器代码。该规则的目的是保持分层架构中清晰的关注点分离，避免业务逻辑与底层存储实现产生紧耦合，从而提升系统的可维护性和可测试性。

## 目标读者
- **应用服务与中间层开发者**：必须理解并遵守本规则，在需要访问数据时使用仓储接口（Repository Interface）或服务抽象层（Service Abstraction），而不直接引用数据库模块。
- **代码审查者**：在审查 Pull Request 时识别并阻止任何违反该规则的直接导入。
- **架构师与基础设施负责人**：该规则是分层边界的重要契约，与仓储模式、领域驱动设计相结合，确保整体架构的长期整洁。

## 规则适用范围
- **适用文件**：项目中所有 `src/**/*.ts` 文件，但排除 `src/db/**` 自身。
- **违规严重性**：`error` 级别，强制检查，不可绕过。
- **支持自动化检查**：可通过 linter 或自定义工具实时拦截违规导入。

## 关键决策与工作流锚点
- **决策锚点**：该规则决定了数据访问层与业务逻辑层的沟通方式。任何新的数据查询或写入操作都应当通过仓储接口定义，而非直接调用数据库驱动或 ORM 方法。
- **工作流锚点**：开发者在实现新功能时，应首先定义或引用仓储接口，然后由数据库层实现该接口。代码审查流程中必须包含对导入来源的审计。
- **例外处理**：没有例外。违反该规则的代码将直接标记为错误，必须修改通过后合入。

## 核心要点
- 禁止在 `src/db/` 以外的任何文件中直接使用 `import ... from 'src/db/...'`。
- 请使用仓储接口（如 `Repository<T>`）或已封装的服务抽象类来访问持久化数据。
- 违规行为被视为架构缺陷，应作为错误在持续集成流程中阻止合并。
