# ArchSpine 配置摘要：`no-direct-db-import` 规则

本架构规则通过禁止应用代码直接导入数据库层，强制维护清晰的领域边界。

## 控制的配置项

该规则由以下关键参数定义：

| 参数 | 值 | 含义 |
|------|-----|------|
| `schemaVersion` | `1.0.0` | 规则模式的版本号；仅当规则结构变更时才需升级。 |
| `ruleId` | `no-direct-db-import` | 工具链（如 lint、CI 检查）引用的唯一标识符。 |
| `title` | 禁止直接导入数据库层 | 看板或报告中显示的人类可读标题。 |
| `summary` | 服务层和中间层不得直接依赖底层数据库适配器。 | 规则的核心原则。 |
| `appliesTo` | `["src/**/*.ts", "!src/db/**"]` | Glob 模式：所有 `src/` 下的 TypeScript 文件，**排除** `src/db/` 目录本身。 |
| `severity` | `error` | 违规会被视为阻断性错误，**阻止构建**及 CI 流水线通过。 |
| `enforceable` | `true` | 该规则由静态分析或 lint 自动校验。 |
| `rationale` | 保持领域边界清晰，避免业务层耦合底层存储实现。 | 规则制定理由。 |
| `bodyMarkdown` | （包含违规示例和允许模式的 Markdown 内容） | 规则的详细文档。 |

## 运维风险与稳定性注意事项

- **稳定性收益：** 执行本规则可防止业务逻辑与数据库实现深度耦合，从而保留重构存储层（例如从 PostgreSQL 切换至其他适配器）的能力，而无需修改应用服务代码。
- **配置风险：** `appliesTo` 的 Glob 模式必须与实际项目结构一致。若 `src/db/` 被重命名或结构发生变化，排除规则可能失效，导致误报。反之，若排除模式过于宽泛，可能错误地阻止某些合法的导入。
- **逐渐退化：** 即使是无意的违规，若反复发生，也会逐渐侵蚀领域隔离，使未来的测试和关注点分离变得极为困难。规则的 `error` 严重级别确保能在 CI 中尽早发现。
- **建议：** 定期审计 `appliesTo` 模式是否匹配当前目录结构，并检查 `bodyMarkdown` 中的示例以保持更新。
