# ArchSpine 镜像系统：索引与同步状态清单

本文档描述了 **ArchSpine 镜像系统** 的索引与同步状态清单文件。该清单记录了哪些源文件已被索引、它们的内容哈希值以及同步状态，用于确保 Spine（骨架）与源仓库之间的一致性。操作员可通过此清单了解索引完整性、同步模式及潜在的稳定性风险。

## 配置控制说明

该清单文件是一个 JSON 结构，主要控制以下方面：

- **索引状态**：记录每个源文件是否已索引（`sourceExists`）、最后索引时间（`lastIndexedAt`）以及对应文档的映射路径。
- **内容哈希**：使用 SHA-256（64 位十六进制字符串）对源文件内容进行哈希，用于检测文件变更，触发重新索引。
- **同步模式**：记录最近一次同步的类型（`full` 或 `incremental`），影响增量变更的应用方式和反向索引的重建逻辑。
- **文档映射**：按语言区域（locale）存储从源文件到转换后文档的路径。

## 重要参数说明

| 参数 | 说明 | 操作注意事项 |
|------|------|---------------|
| `schemaVersion` | 清单格式版本号，必须与生成器兼容。 | 版本不匹配会导致解析错误，索引可能无法加载。 |
| `sync.lastSyncMode` | 最近一次同步模式（`full` 或 `incremental`）。 | `incremental` 模式下，若反向索引不完整，可能遗漏变更。全量同步后建议检查 `reverseIndexComplete`。 |
| `sync.reverseIndexComplete` | 布尔值，指示从转换产物到源文件的反向映射是否完整。 | 若为 `false`（如示例中所示），则无法完整追溯，可能影响交叉引用和搜索功能。建议在全量同步后立即重建反向索引。 |
| `sync.indexedUnitCount` | 当前已索引的文件数量。 | 必须等于 `sourceExists: true` 的条目数。若不一致，说明索引状态与实际文件不符，需重新扫描。 |
| `files[].contentHash` | 源文件内容的 SHA-256 哈希值。 | 用于变更检测与缓存失效。若哈希过时或缺失，系统可能跳过必要的重新索引，导致文档与源码不一致。 |

## 稳定性与风险提示

- **内容哈希必须准确**：如果 `contentHash` 是旧值或错误值，系统将无法检测到源文件变更，导致文档更新遗漏。因此，任何手动修改清单时务必保持哈希与文件实际内容一致。
- **`reverseIndexComplete` 为 false 的风险**：反向索引不完整会破坏从生成文档到原始源文件的追溯能力，降低搜索和交叉引用可靠性。应在每次索引变更后（尤其是增量同步后）触发重建。
- **清单与文件系统的同步**：该清单必须与实际的源文件状态保持一致。手动编辑清单极其危险，容易使索引记录与文件系统脱节，造成无声的故障（如索引以为文件存在但实际已删除，或反之）。
- **索引数量校验失败**：若 `indexedUnitCount` 与 `sourceExists` 计数不符，应视为严重不一致，需立即恢复或重新生成索引。
