# ArchSpine 共享定义 – 配置概要

此基础模式定义了 ArchSpine 所有组件和模式中使用的可重用数据类型与验证规则，确保格式一致性，并作为版本字符串、时间戳、路径、哈希值、语言代码及枚举分类等的唯一来源。

## 配置控制的内容

该模式控制 ArchSpine 生态系统中每个共享字段的允许格式和值。所有进入系统的数据必须符合这些定义，否则验证失败。关键控制点包括：

- **版本号**：所有 `schemaVersion` 字段必须遵循 `主版本.次版本.补丁` 的语义化版本格式。格式错误会破坏整个模式验证链。
- **时间戳**：`isoTimestamp` 强制使用 ISO 8601 格式，对事件排序、缓存管理和增量处理至关重要。
- **文件路径**：`repoRelativePath` 和 `scopePath` 确保路径为相对路径、跨平台安全，并阻止绝对路径、反斜杠或双斜杠（防止路径遍历攻击）。
- **内容完整性**：`contentHash` 要求恰好 64 个小写十六进制字符（SHA-256），对缓存一致性和损坏检测至关重要。
- **语言与区域**：`locale` 验证 BCP 47 标签；错误标签会导致本地化查找失败。`languageSelectionKey` 的非空检查可防止 atlas 合并时缺失键。
- **枚举类型**：`sourceLanguage`、`fileKind`、`symbolKind`、`dependencyRelation`、`edgeProvenance` 和 `pipelineStage` 都被限制为封闭枚举集合。任何未知值都会被拒绝，除非显式处理（例如，源代码语言中的 `"unsupported"`）。

## 操作员需关注的参数

| 参数 | 重要性说明 |
|------|------------|
| `schemaVersion` | 版本不匹配可能引发级联问题；必须与预期模式的主版本一致。 |
| `repoRelativePath` | 安全性关键：阻止恶意路径。误用可能导致合法路径被误拒——需定期审查正则表达式。 |
| `contentHash` | 完整性锚点；过时或错误的哈希会使下游缓存和比较失效。 |
| `locale` | 格式错误在某些工具中可能静默失败；确保所有语言条目严格遵循 BCP 47。 |
| `sourceLanguage` | 添加新语言需要更新模式；在此之前，新语言必须视为 `"unsupported"`。 |
| `dependencyRelation` | 将 `type-import` 误认为 `import` 会扭曲依赖权重计算——对依赖分析和治理至关重要。 |
| `pipelineStage` | 过时或错误的阶段记录可能导致下游消费者无效（例如，期望 `"llm"` 数据却使用了 `"ast"` 数据）。 |
| `edgeProvenance` | 影响依赖图中的置信度评分。过度依赖 `"inferred"` 边可能误导分析结果。 |

## 稳定性关注点与操作风险

- **紧密耦合**：对该模式的任何更改都会传播到所有依赖的模式和运行时验证器。向后不兼容的更改（如添加必需字段、收紧正则表达式或移除枚举值）需要**主版本号升级**。
- **枚举刚性**：封闭枚举提供了强验证，但支持新工具或语言需要发布模式更新。权衡取舍：放宽为 `"type": "string"` 并辅以文档指导能增加灵活性，但会失去自动验证能力。
- **安全性与可用性**：严格的路径和哈希模式可防止注入和路径遍历攻击，但可能拒绝合法的边缘情况（例如包含非 ASCII 字符的路径、新的语言代码格式）。建议定期审计并收集用户反馈。
- **缓存与排序风险**：时间戳和哈希不匹配可能破坏增量构建和依赖缓存。操作员应确保时间戳在 UTC 时区以一致的精度生成。

生产环境中应将此模式视为不可变，除非规划了协调升级。通过模式版本协商避免静默失败。
