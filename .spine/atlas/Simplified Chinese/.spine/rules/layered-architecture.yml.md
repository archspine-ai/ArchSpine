# ArchSpine 分层架构约束

## 目的
本文档旨在明确定义 ArchSpine 镜像系统分层模型（CLI → 服务 → 核心/任务/引擎 → 基础设施）的架构边界规则。它确保每个层保持在各自的责任范围内，防止架构退化，并保证运行时行为保持解耦且可测试。开发者与贡献者依据这些约束来维护整个代码库的关注点分离原则。

## 受众
本文档面向：
- **开发者和贡献者**：需要理解并强制执行分层规则，以便在添加或修改代码时保持边界清晰。
- **代码审查者和架构师**：在验证拉取请求时，对照已定义的分层规则进行检查。
- **新入职成员**：希望通过本文档快速了解系统的组织方式。

## 本文档锚定的决策与工作流
- **拉取请求审查**：以规则为检查清单，确保每个层不越界。
- **架构决策**：关于新功能放置位置的决策，将参考本文档中的范围与约束。
- **重构工作**：参考本文档以保持分层模型，避免重新引入耦合。

## 关键要点
1. **CLI 入口点**必须保持轻量——不得包含流水线或持久化逻辑。
2. **服务层**可编排跨层工作，但不得嵌入基础设施细节。
3. **核心模块**定义流水线契约和共享状态，不依赖 CLI 代码。
4. **任务阶段**基于核心契约和引擎实现阶段本地工作。
5. **引擎**提供可复用的分析和转换逻辑，不涉及展示层关注点。
6. **基础设施外观**暴露稳定的底层能力，必须避免吸纳编排逻辑。

## 约束范围
规则覆盖所有层：CLI、服务、核心、任务、引擎和基础设施。每条规则包含**严重级别**（错误或警告）和**理由**。错误表示违反架构边界；警告标识应避免的模式以保持清晰性。

## 不在范围内
本文档**不**涉及：
- 具体模块的实现细节。
- 部署或操作流程。
- 代码风格或格式化指南。
- 版本控制或发布流程。
