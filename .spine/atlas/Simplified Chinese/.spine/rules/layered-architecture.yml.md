# ArchSpine 架构约束规则手册

## 目的
本文档为 ArchSpine 项目建立并强制执行分层架构约束。它清晰定义了 CLI 入口、服务编排、核心流水线契约、任务阶段、分析引擎以及基础设施外观之间的边界。目标是防止架构漂移，确保每一层专注于自身职责，使系统在演进过程中保持模块化、可测试和可维护。

## 背景与受众
本文档面向 ArchSpine 仓库的所有贡献者，包括开发者、代码审查者和架构师。它既是用以理解预期模块结构的参考手册，也是在开发和代码审查过程中强制执行导入和职责边界的规则手册。新贡献者应首先阅读本文档，以了解其变更应归属于何处。

## 关键要点
- 项目遵循严格的分层架构：CLI -> services -> core/tasks/engines -> infra，依赖方向明确。
- CLI 模块必须保持为薄入口，不得吸收流水线或持久化逻辑。
- 服务负责编排运行时行为；视图相关的服务应位于 `src/services/view/` 下。
- 核心模块定义流水线契约，且不得依赖 CLI 代码。
- 引擎提供可复用的分析逻辑，必须与服务编排解耦。
- 基础设施模块暴露稳定的外观接口，不得吸收编排职责。
- 每条约束都有严重级别（错误或警告）和明确的理由说明。

## 不涵盖范围
- 具体模块的实现细节
- 外部 API 契约或面向用户的文档
- 部署或基础设施配置说明
- 测试策略或 CI/CD 流水线配置
