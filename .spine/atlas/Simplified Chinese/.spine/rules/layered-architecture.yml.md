# ArchSpine 分层架构约束

## 目的
本文档定义了一套分层架构约束，强制 CLI、services、runtime、core、tasks、engines 和 infra 模块之间保持清晰的关注点分离。其目标是通过防止不恰当的跨层依赖，使代码库保持可维护性、可测试性和可理解性。

## 目标读者
面向为 ArchSpine 仓库做出贡献的开发者和 AI 代理。这些规则定义了添加或修改代码时必须遵守的架构原则，并将通过静态分析或代码审查来执行，以确保项目演进过程中预期的模块化结构得以保留。

## 关键架构决策
- **CLI 模块**必须保持为薄适配器，不得吸收流水线或持久化逻辑。
- **Services 层**编排 core/tasks/engines/infra 模块，但与视图相关的服务应位于 `src/services/view/` 下。
- **Runtime 兼容性模块**集中管理 schema 版本分类和确定性内存迁移。
- **核心（core）模块**定义流水线契约，不依赖 CLI 入口代码。
- **任务（tasks）模块**在核心契约和引擎之上实现阶段局部工作。
- **引擎（engines）**提供可复用的分析和转换逻辑，独立于 CLI 和服务编排。
- **基础设施（infra）模块**暴露稳定的底层外观，调用者应优先使用公共外观而非深入内部实现路径。

## 锚定的工作流
- **代码审查**：审查者检查新代码是否违反依赖方向（例如 CLI 导入 core 内部模块）。
- **静态分析**：通过 linting 或架构验证工具（如 dependency-cruiser）自动执行这些规则。
- **架构重构**：在演进代码库时，这些约束指导新模块的放置位置以及如何重构现有结构。
- **贡献者入职**：新团队成员通过本文档了解预期的层边界。
