# ArchSpine 分层架构约束

## 目的
本文档为 ArchSpine 项目建立并强制执行分层架构约束。它清晰定义了 CLI、services、core、tasks、engines 和 infra 各层之间的边界，确保每层都履行其规定职责：CLI 仅作为薄入口，services 负责编排，core 定义流水线契约，tasks 实现阶段本地工作，engines 提供分析转换，infra 提供底层能力。

## 读者对象
面向所有参与 ArchSpine 代码库的开发者。本文档作为理解项目架构愿景的参考，帮助防止架构漂移，并在开发与代码评审中指导代码的合理放置。

## 关键架构约束
- **CLI 入口隔离（错误）**：CLI 模块必须保持轻量，不能吸收下层流水线或持久化逻辑，使运行时行为可脱离 CLI 复用。
- **服务运行时边界（警告）**：Services 可编排 core/tasks/engines/infra，但视图相关服务代码应位于 `src/services/view/` 下，保持结构清晰。
- **核心流水线隔离（错误）**：Core 模块定义流水线契约，不依赖 CLI 入口代码，确保调用方式不影响稳定性。
- **任务阶段边界（警告）**：Tasks 基于 core 契约和 engines 实现阶段本地工作，不得接管 CLI 解析或无关编排。
- **引擎独立性（错误）**：Engines 提供可复用分析逻辑，不导入 CLI 入口，与服务编排解耦。
- **基础设施门面导入（警告）**：优先使用公有 infra 门面，而非深入私有实现路径，使重构更安全。

## 工作流锚点
本文档锚定开发与代码评审中的代码放置决策。标记为“错误”的违规必须修复以通过 CI；“警告”违规需审查架构一致性。分层模型指导代码库演进，确保新模块正确放置，依赖保持无环。
