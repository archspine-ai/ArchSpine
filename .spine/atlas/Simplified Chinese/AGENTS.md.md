# ArchSpine 项目约定

## 目的
本文档是 ArchSpine 仓库所有贡献者的核心参考指南。它确立了项目结构、编码风格、测试、文档以及如何利用内置的 ArchSpine 上下文系统进行高效开发的基本规则。

## 背景与受众
目标读者是任何在 ArchSpine 代码库上工作的开发者或维护者。对于需要了解项目约定的新贡献者，以及为确保所有贡献一致性的团队成员来说，本文档尤为重要。此外，本文档还为 AI 代理提供了关于如何与 ArchSpine 上下文文件交互的关键指令。

## 主要职责
- 描述项目目录布局和模块组织
- 列出构建、测试和开发命令
- 指定 TypeScript 编码风格和命名约定
- 概述使用 Vitest 的测试指南
- 定义提交和拉取请求的约定
- 解释文档维护以及临时工作区的作用
- 提供使用 ArchSpine 上下文系统的说明

## 不涵盖范围
- 详细的 API 参考或函数级文档
- 架构设计原理或规划文档
- 第三方依赖许可证或安全策略
- 部署或发布流程

## 关键要点
- 项目遵循 `src/` 下的严格模块化结构，职责划分清晰（CLI、核心、服务、基础设施）。
- 所有代码必须遵守定义的 TypeScript 风格：ES 模块、严格类型、2 空格缩进以及特定的命名约定。
- 所有行为变更都必须进行测试，使用 Vitest，研究用的测试夹具应与主测试套件分开存放。
- 提交信息必须遵循 Conventional Commits 格式，拉取请求必须包含验证步骤和相关问题链接。
- 文档变更必须保持英文和中文版本对齐；临时工作区（`docs/temporary-to-be-cleared/`）是临时变更记录的唯一真实来源。
- AI 代理和开发者在进行广泛搜索前，必须先查阅 ArchSpine 上下文（`.spine/atlas/`、`.spine/view/pages/`），并且不得直接修改 ArchSpine 生成的输出文件。
