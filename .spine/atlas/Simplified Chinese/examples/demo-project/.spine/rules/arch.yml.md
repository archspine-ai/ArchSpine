# ArchSpine 演示项目架构规则

## 文档目的

本文档为 ArchSpine 演示项目制定了两条基础规则，以确保清晰的架构层次和可读的领域契约。它锚定了强制层隔离和文档标准化这两个关键决策，直接影响项目中每个 API 处理器和领域类的编写方式。

## 目标读者

- **开发者**：在添加或修改 API 处理器（`src/api/`）时，必须理解导入限制。
- **开发者**：在 `src/domain/` 中编写领域类时，需遵循 TSDoc 注释要求。
- **审查者与架构师**：根据该文档验证代码是否遵守项目规则。

## 为何制定这两条规则

- **层隔离** 防止 API 层直接依赖基础设施层的具体实现，保持代码库的灵活性和架构的清晰性。若无此规则，处理器可能绕过领域层，将表现逻辑与底层细节耦合。
- **文档标准** 确保公共领域契约在自动生成的架构视图中依然可读且自解释，降低人工和工具解读的歧义。

## 关键要点

1. **API 处理器不得直接导入基础设施模块**（`src/infra/`），应通过领域层进行交互。
2. **`src/domain/` 下的所有公共类和方法应包含 TSDoc 注释**，以保持架构契约的清晰且可维护。

## 决策锚点

- **层隔离** 规则以 **错误** 级别强制执行：`src/api/` 中任何直接导入 `src/infra/` 的代码必须被修正。
- **文档** 规则以 **警告** 级别强制执行：缺少 TSDoc 注释的公共领域成员应予以补充，以保证一致性。
