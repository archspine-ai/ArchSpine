# ArchSpine 同步运行记录

## 角色
此文件是 ArchSpine 镜像系统中**单次同步运行的执行记录**。它捕获同步操作的完整生命周期、状态和诊断元数据。

## 主要职责
- 跟踪同步操作的生命周期和状态（待处理 → 运行中 → 已完成/失败）。
- 记录同步运行的元数据：是否为完整同步或增量同步、是否由钩子触发、以及任何恢复状态。
- 记录每个阶段的进度和完成时间戳。
- 在扫描清理阶段捕获过滤后的文件路径和受影响的目录。
- 记录每个源文件的 AST 提取结果。
- 记录每个配置文件或文档文件的摘要生成结果。

## 重要不变性
- `status` 字段必须为以下之一：`pending`（待处理）、`running`（运行中）、`completed`（已完成）、`failed`（失败）。
- `runId` 在每次执行中必须唯一。
- 所有阶段条目必须具有与运行生命周期一致的状态和时间戳。
- `metadata.full` 标志决定同步是完整运行还是增量运行。
- `metadata.hookMode` 标志在正常运行时必须为 `false`；`true` 表示由钩子触发的运行。

## 负面范围 / 超出范围
此记录是**诊断性工件**，而非配置文件。它不会直接影响系统行为。但是，如果运行记录损坏或丢失，系统可能无法恢复中断的同步或审计过去的操作。

## 最重要的导出/外部可见行为
- `resumedFromRunId` 和 `resumeCandidateCount` 字段对恢复至关重要；不正确的值可能导致系统尝试从无效状态恢复。
- 扫描清理阶段中的 `filteredFiles` 列表决定了哪些文件会被处理；空列表或不正确的列表可能导致关键文件被跳过。
- AST 提取和摘要生成阶段中的每个文件状态允许系统检测部分失败；如果某个文件标记为 `failed`，系统应重试或发出警报。

## 稳定性和风险
此运行记录是诊断性工件，而非配置文件。它不会直接影响系统行为。但是，如果运行记录损坏或丢失，系统可能无法恢复中断的同步或审计过去的操作。`resumedFromRunId` 和 `resumeCandidateCount` 字段对恢复至关重要；如果它们不正确，系统可能尝试从无效状态恢复。scan-cleanup 中的 `filteredFiles` 列表决定了哪些文件会被处理；如果列表为空或不正确，关键文件可能被跳过。ast-extraction 和 summarization 中的每个文件状态允许系统检测部分失败；如果某个文件标记为 `failed`，系统应重试或发出警报。
