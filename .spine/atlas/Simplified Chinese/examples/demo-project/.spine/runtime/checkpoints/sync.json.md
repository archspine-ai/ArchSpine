# ArchSpine 同步管线检查点配置摘要

本文档总结了 ArchSpine 同步管线使用的执行检查点记录。与改变系统行为的传统配置文件不同，此检查点文件是特定 sync 命令运行的权威追踪记录。它记录了启动、进度和最终状态，包括每个阶段的时间戳和每个文件的完成状态。运维人员依赖它进行恢复、调试和审计。

## 该配置控制的内容

检查点本身不控制管线行为，而是记录管线执行的结果。然而，其中的参数决定了系统如何解释过往运行以及能否从中断中恢复。关键参数如下：

- **schemaVersion**：必须为 `"1.0"`。任何偏差都会导致运行时误读或拒绝检查点。
- **command**：标识生成此检查点的 CLI 命令（例如 `"sync"`）。这告诉运行时期望哪些管线阶段以及如何在恢复时重新进入流程。
- **runId**：唯一字符串（通常基于时间戳），用于将检查点关联到特定执行。用于防止重复处理并通过 `resumedFromRunId` 支持恢复。
- **status**：运行的整体状态——`"completed"`、`"failed"`、`"running"` 或 `"pending"`。此标志驱动下游逻辑：`"failed"` 状态可能触发修复/恢复工作流；`"running"` 可能阻止新运行。
- **startedAt / updatedAt**：ISO 8601 时间戳，标记运行开始时间和检查点最后修改时间。对基于时间的诊断、检测过时检查点以及确保并发访问下的一致性至关重要。
- **metadata**：包含上下文标志：
  - `full` – 是全量同步还是增量同步。
  - `hookMode` – 是否由 git 钩子触发。
  - `resumedFromRunId` – 被继续的先前运行。
  - `resumeCandidateCount` – 找到的可行恢复点数量。零计数可能表示全新开始或无法定位先前检查点。
- **stages**：管线阶段名称（例如 `"Reconcile State & Cache"`、`"scan-cleanup"`、`"ast-extraction"`、`"summarization"`）到其状态和时间的映射。每个阶段记录自己的 `status`、`startedAt`、`completedAt` 和 `updatedAt`。`scan-cleanup` 阶段可能包含 `data`（过滤后的文件和受影响的目录）。`ast-extraction` 和 `summarization` 阶段包含 `items`——文件级完成对象（状态和时间戳）的映射。阶段顺序执行；失败的阶段会阻止后续阶段启动。

## 操作风险与稳定性注意事项

此文件是关键运行时工件。其完整性直接影响系统以下能力：

- 恢复中断的运行而不跳过或重复工作
- 提供准确的诊断信息
- 避免重复处理或错误报告状态

**损坏或不一致**是主要风险。如果检查点文件损坏或与实际管线进度不同步（例如由于磁盘故障或非原子写入），后续运行可能跳过阶段、重复处理已完成的工作，或错误报告状态。**切勿手动编辑此文件**；运行时必须始终原子地更新它。

**并发与写入争用**：高写入并发可能导致部分更新。运行时应使用文件锁或原子写入模式。磁盘故障构成最大危险。

**对先前运行的依赖**：`resumedFromRunId` 和 `resumeCandidateCount` 字段创建了依赖链。如果引用的 `runId` 丢失或其检查点损坏，恢复将静默失败——系统可能启动全新运行或出现异常行为。运维人员应监控先前检查点是否仍然可访问且有效。

**所有时间戳必须是有效的 ISO 8601** 并满足每个阶段和项目的 `startedAt ≤ completedAt`。违反此条件可能混淆诊断，并表明存在错误或数据损坏。

总之，在正常操作期间将检查点文件视为只读记录，由运行时原子更新，并定期备份以防范上述风险。
