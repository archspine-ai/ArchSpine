# ArchSpine 同步运行记录 – 配置摘要

本文档说明 ArchSpine 镜像系统在执行同步操作后生成的运行记录的结构和含义。运维人员可借助这些记录验证同步是否成功、调试故障以及了解系统行为。

## 配置项及其作用

`command` 字段定义执行的操作。当其值为 `"sync"` 时，运行完整的同步流水线。`status` 字段显示最终结果：`"completed"` 表示流水线正常结束；非完成状态（如 `"failed"`、`"aborted"`）表明出现问题，需要在后续操作前进行调查。

`metadata` 部分包含三个关键开关：

- **`full`** – 若为 `true`，系统执行全量状态回放而非增量更新。全量同步资源消耗较大，但能捕获所有变更。运维人员应注意频繁的全量同步可能影响性能。
- **`hookMode`** – 若为 `false`，该运行由手动或程序触发；若为 `true`，则由文件系统钩子事件触发。此信息用于审计以及决定是否需要重新触发钩子。
- **`resumedFromRunId`** – 若存在，表示本次运行续接了先前某次运行（如崩溃后恢复）。这支持崩溃恢复并避免重复处理。运维人员需确认续接运行的开始时间晚于原运行时间，并留意是否存在文件遗漏。

`stages` 对象记录每个流水线阶段：状态协调、清理、AST 提取和汇总。每个阶段有自己的时间戳和状态。任一阶段失败将终止整个运行。AST 提取和汇总下的 `items` 列出实际处理的文件及其个体状态。

## 操作风险与稳定性说明

- **恢复逻辑** – `resumedFromRunId` 的存在表明系统具备中断恢复能力。但如果恢复逻辑有缺陷，可能出现文件遗漏或重复处理。务必对照期望文件列表进行交叉校验。
- **全量同步资源开销** – `full: true` 表示完整状态扫描，安全但可能对 I/O 和内存造成较大压力。在全量同步运行时注意监控系统负载。
- **阶段失败风险** – 由于流水线在任一阶段失败时即中止，某个阶段缓慢或异常会阻塞整个同步。请关注各阶段时间戳，留意异常耗时。
- **过滤文件列表** – `scan-cleanup` 阶段输出 `filteredFiles` 列表，用于审计和后续处理。若过滤规则有误，可能遗漏重要文件。需验证该列表包含了所有应处理的文件且只排除了有意跳过的文件。
- **时间顺序校验** – 每个阶段的 `startedAt` 必须早于 `completedAt`，且阶段需按顺序执行。任何时间戳重叠或乱序都可能是 Bug 或数据损坏的信号。

该运行记录为同步过程提供了完全透明的追踪信息，是监控系统健康状态和诊断异常的主要依据。
