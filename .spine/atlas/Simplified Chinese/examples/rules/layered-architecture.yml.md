# ArchSpine 分层架构约束

## 为什么需要这份文档

本文档为 ArchSpine 项目定义了分层架构约束，确保高层模块不会依赖低层模块，领域层保持纯粹且独立。它明确了领域隔离、仅接口依赖和 API 层分离的具体规则，从而避免循环依赖、业务逻辑脆弱以及难以自动验证的问题。

没有这些约束，系统将面临难以重构和维护的风险。

## 谁应该阅读

- **开发者**：编写或审查 `src/domain/`、`src/api/`、`src/infra/` 或 `src/cli/` 中任意模块代码的人员
- **架构师**：设计新功能或重构现有组件的人员
- **QA 与运维工程师**：设置静态检查规则或自动化架构验证的人员

## 它如何锚定决策和工作流

本文档中的规则应用于以下环节：

- **代码审查**：每个拉取请求必须符合这些依赖方向
- **静态分析与检查**：自动化工具（如 dependency-cruiser、ESLint 导入规则）将违规标记为错误或警告
- **架构验证流水线**：CI/CD 步骤拒绝破坏分层模型的提交

## 关键要点

- **领域隔离**（错误级别）：`src/domain/` 内的模块绝不能导入 `src/infra/` 或 `src/cli/` 的内容。领域层必须保持纯粹。
- **仅接口依赖**（警告级别）：`src/domain/` 模块只能依赖其他领域逻辑或通用工具，保持关注点集中。
- **API 层分离**（错误级别）：`src/api/` 内的模块只能调用服务层或领域层，不得绕过业务逻辑。

理解并应用这些规则，能够让镜像系统以清晰的边界演进，使未来的变更更加安全且可预期。
