# ArchSpine Configuration Summary

## Role
The `package.json` file defines the project metadata, dependency graph, executable entry points, and runtime requirements for the ArchSpine mirror governance system. It orchestrates the dependency tree for static analysis (AST-grep), LLM integration (Google Gemini, OpenAI), persistent storage (better-sqlite3), CLI interaction, and documentation generation. It also configures the build, test, and release pipeline via devDependencies (TypeScript, ESLint, Vitest, VitePress), declares two CLI binaries (`archspine` and `spine`), and sets the minimum Node.js engine version (>=20) to ensure core API compatibility.

## Key Parameters

- **`name`**: `archspine` — Uniquely identifies the package in the npm ecosystem; used by package managers and registries.
- **`version`**: `1.0.0` — Initial release version. Semantic versioning must be maintained to signal breaking changes.
- **`bin`**: `archspine` and `spine` — Two CLI entry points that must point to the compiled JavaScript entry; any misconfiguration breaks the entire command-line interface.
- **`engines`**: `node >=20` — Gatekeeping the runtime; if a user attempts to run on an older Node.js, npm will warn or refuse, enforcing system requirements.
- **`dependencies`**: Production dependencies such as `@ast-grep/*`, `better-sqlite3`, `@google/generative-ai`, `openai`, `undici`, etc. These directly affect functionality; version ranges must be locked to avoid regressions in parsing, database, or AI connectivity.
- **`devDependencies`**: TypeScript, ESLint, Vitest, VitePress — These control development workflow and documentation building. Mismatched versions can break CI, build, or test pipelines.
- **`lockfileVersion`**: `3` — Ensures deterministic installs across environments. Changing this could allow dependency resolution drift, leading to "works on my machine" bugs.

## Stability and Risks

This configuration file is the root dependency manifest. Any misconfiguration (e.g., incorrect version ranges, missing lockfile, engine mismatch) can cause complete build failures, runtime crashes, or security vulnerabilities from unvetted transitive dependencies. The fixed engine requirement (`node >=20`) prevents silent API breakage. The dual binary entry points must remain consistent; renaming or removing them silently breaks the CLI user experience. Regular dependency updates are critical — especially for security-critical packages like `undici`, `node-fetch`, and the LLM SDKs. The devDependencies for documentation (VitePress) and testing (Vitest) affect the reliability of documentation generation and test suites, which in turn impacts developer confidence and release quality.
