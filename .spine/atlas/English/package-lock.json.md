# ArchSpine Package Configuration

## Role
Defines the runtime dependencies and build toolchain for the ArchSpine project, a mirror system that uses AST-based code analysis and AI integration.

## Key Responsibilities
- **Package dependency management** for AST parsing across multiple languages (C, C++, Go, Java, Python, Rust)
- **AI model integration** via Google Generative AI and OpenAI SDKs
- **Local database storage** using better-sqlite3
- **CLI tool entry points** (`archspine`, `spine`)
- **Development tooling**: linting (ESLint), formatting (Prettier), testing (Vitest), documentation (VitePress), and TypeScript compilation

## Notable Invariants
- Node.js engine must be >= 20
- All production dependencies must be resolved with integrity hashes
- Lockfile version must be 3 for reproducible installs
- License must remain Apache-2.0

## Negative Scope
This configuration does not define any runtime behavior, application logic, or system architecture. It solely manages the dependency graph and build toolchain.

## Exported Behavior
- **`dependencies`**: All production packages required at runtime, including AST parsers for six languages, AI SDKs, SQLite storage, and CLI utilities.
- **`devDependencies`**: Build and development tools such as TypeScript, ESLint, Prettier, Vitest, and VitePress for documentation.
- **`engines`**: Specifies the minimum Node.js version (>=20) required to run the project.
- **`bin`**: Defines CLI entry points `archspine` and `spine` that map to the compiled JavaScript file.
- **`lockfileVersion`**: Ensures deterministic dependency resolution using lockfile version 3.

## Stability and Risks
This configuration directly impacts system stability by controlling which third-party libraries are available at runtime. Incorrect version pinning or missing integrity checks could lead to supply-chain vulnerabilities or runtime failures. The broad set of AST parsers increases the attack surface, and reliance on external AI APIs introduces network dependency risks. The lockfile ensures reproducible builds, but any manual modification to package.json without updating the lockfile may cause inconsistent environments.
