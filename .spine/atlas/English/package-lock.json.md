# ArchSpine Package Lockfile Configuration Summary

## Overview
This file (`package-lock.json`) is the authoritative lockfile for the ArchSpine project. Its purpose is to lock every direct and transitive dependency to an exact version, guaranteeing reproducible dependency resolution and deterministic builds across all environments. The lockfile enforces integrity checks via hashes and ensures that only packages satisfying the project's engine requirements are installed.

## Key Parameters
- **`lockfileVersion`**: Set to `3`. This modern format supports workspaces, deterministic installs, and is required by npm v7+. Using an older npm version will fail to parse it.
- **`requires`**: Boolean flag set to `true`, indicating the root package (`archspine`) has declared dependencies that must be resolved.
- **`engines`**: Defines the required Node.js version range as `>=20.18.1 <21 || >=22`. If the Node.js runtime does not match this constraint, npm will refuse to install or run the package. This is a hard gate and a common cause of deployment failures on legacy Node.js versions.
- **`bin`**: Maps the CLI commands `archspine` and `spine` to `dist/cli/index.js`. These are the global entry points after installation.

## Notable Dependencies
- **`@ast-grep/lang-c`** (and related `@ast-grep/lang-*` packages): AST parsers for language analysis – critical for code understanding features.
- **`@google/generative-ai`**: Generative AI SDK – powers AI-driven interactions.
- **`better-sqlite3`**: SQLite binding – used for local caching and persistent state.
- **`vitepress`** (dev only): Documentation site generator; not shipped to production.
- **`vitest`** (dev only): Unit test runner; ensures code quality via automated tests.

## Stability & Risks
This lockfile eliminates version drift between environments – identical dependency trees are guaranteed. However, failure to regenerate the lockfile after adding or updating dependencies will cause `npm install` to fail with integrity mismatch errors. Security risks arise from outdated packages containing known vulnerabilities; regular `npm audit` and dependency updates are essential. The **engines** constraint enforces a specific Node.js range; deployment on any unsupported version (e.g., Node.js 18 or 21.x) will be blocked entirely. Operators should validate their runtime before deployment and consider pinning Node.js with `.nvmrc` or Docker.
