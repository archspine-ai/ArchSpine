The `schemas/examples` directory defines the core schema framework for the ArchSpine mirror system. It provides the foundational data models for metadata indexing, configuration, and governance. This directory contains six key schema files that together establish a unified validation, provenance tracking, and policy enforcement system.

Notable children:

- **`shared.schema.json`**: Provides reusable base type definitions (e.g., version strings, timestamps, file paths, content hashes) to ensure type consistency across all schemas.
- **`spine-folder-unit.schema.json`**: Defines the schema for directory-level metadata, capturing role, responsibility, child file entries, and provenance (index timestamp, generator version, pipeline stages).
- **`spine-manifest.schema.json`**: Records the synchronization state of indexed source files, their documentation mappings per locale, and integrity verification via content hashes. Tracks sync mode (full/incremental), reverse index completion, and total unit count.
- **`spine-project-unit.schema.json`**: Structures project metadata, module decomposition (directory, role, child count), and generation provenance.
- **`spine-rules.schema.json`**: Governs architectural and coding rule definitions with fields for ID, title, severity, enforceability, applicability (`appliesTo`), and a Markdown body for human-readable guidance.
- **`spine-unit.schema.json`**: The overarching schema for SpineUnit documents, enforcing required fields like `schemaVersion`, `identity`, `semantic`, `skeleton`, `graph`, and `provenance`, along with nested structures for invariants and change intent.

These schemas are tightly integrated to support the ArchSpine system's core responsibilities: indexing directory structures and file metadata, synchronizing documentation across locales, enforcing architectural rules, and tracking the provenance of all indexing and generation processes. The shared schema acts as the foundation, while the specialized unit schemas cover folder, project, manifest, and rule domains. Together, they enable consistent validation, incremental updates, and integrity checks across the entire mirror system. The most critical implementation areas are validation consistency, provenance tracking for indexing and generation, policy enforcement via rules, and synchronization integrity management.
