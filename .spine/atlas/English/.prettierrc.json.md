# ArchSpine Configuration Summary: Code Formatting (Prettier)

This configuration defines code formatting rules for the project using Prettier, ensuring consistent style across the codebase. Operators should review the settings to align with team conventions and CI expectations.

## Current Values

| Parameter        | Value  |
|------------------|--------|
| `printWidth`     | 100    |
| `singleQuote`    | `true` |
| `semi`           | `true` |
| `trailingComma`  | `"all"`|

## Parameter Descriptions

- **`printWidth`** (100) — Specifies the maximum line length before wrapping. Higher values reduce line breaks but may cause horizontal scrolling. Here set to 100, which is wider than the default 80. Teams should ensure editors and review tools can handle longer lines.

- **`singleQuote`** (`true`) — When true, prefers single quotes over double quotes for strings. Reduces visual noise in codebases that adopt this convention. Consistent with many JavaScript/TypeScript projects.

- **`semi`** (`true`) — When true, ensures semicolons are added at the end of statements. Promotes consistency especially in mixed JS/TS projects. Turning this off can lead to ASI (automatic semicolon insertion) bugs.

- **`trailingComma`** (`"all"`) — Controls trailing commas in object/array literals. `"all"` adds them wherever possible, enabling cleaner diffs in version control (e.g., adding a new property changes only one line). The value must be one of `"none"`, `"es5"`, or `"all"`.

## Stability and Risks

- **Risk level**: Low. The configuration uses standard Prettier options and valid values.
- **Failure modes**: Any invalid value (e.g., `trailingComma: "everything"`) will cause Prettier to error and block formatting or CI pipelines.
- **Operational notes**:
  - The higher `printWidth` (100) may result in longer lines during code review — ensure your team's review tools and monitor widths are comfortable.
  - Changing `semi` to `false` can introduce subtle bugs due to ASI; this setting is safe.
  - `trailingComma: "all"` is widely adopted in modern JS/TS projects and is low risk.

## Recommendations

- Validate this file with Prettier's `--check` flag in CI to catch misconfigurations early.
- If contributing to a broader ecosystem (e.g., open source), ensure consistency with community conventions.
- No runtime impact; this is purely a development tool setting.
