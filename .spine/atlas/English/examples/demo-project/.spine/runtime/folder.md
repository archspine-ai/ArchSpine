# ArchSpine Runtime Directory

The `runtime` directory within the ArchSpine mirror system stores all execution checkpoint records for synchronization runs. It serves as the operational memory of the system, tracking every sync operation's lifecycle, status, and detailed metadata.

## Structure

- **`checkpoints/`** — The sole subdirectory, containing individual checkpoint files for each sync run. Each checkpoint records:
  - Sync type (full vs. incremental)
  - Hook mode
  - Resume state
  - Per-stage progress and completion timestamps
  - Filtered file paths and affected directories during scan-cleanup
  - AST extraction results per source file
  - Summarization results for configuration and documentation files

## Implementation Areas

This directory is critical for:
- **Sync lifecycle management**: Enabling resumable and incremental syncs by persisting state between runs.
- **Audit & debugging**: Providing a complete history of sync operations for forensic analysis.
- **Pipeline integration**: The `provenance` block indicates that these records are generated by the `archspine/1.0.0` generator and pass through `ast` and `llm` stages, meaning checkpoint data is enriched with both structural and semantic insights.

## Concrete Submodules

- **checkpoints/**: Each file inside this folder corresponds to a single sync execution, with a naming convention likely tied to a timestamp or run ID. The data stored here is reusable for resume operations or post-hoc analysis.

For further exploration, refer to the sibling directories such as `ast` (AST extraction results) and `config` (project configuration) which interact with these runtime records.
