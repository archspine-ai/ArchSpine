The `types` directory is the ArchSpine mirror system's contract hub, providing all core type definitions and shared interfaces that enforce consistent data structures across the entire project. It defines the data model for mirror units (SpineUnit, SpineIdentity, SpineSemantic, SpineSkeleton, ArchSpine dependency graph, SpineProvenance, SpineFolderUnit, SpineProjectUnit), the project's central configuration schema (SpineConfig), language support metadata (LanguageSnapshot, LanguageSupport, LanguageDelta), synchronization manifests and DTOs (SpineManifest, SyncBlock, DocRef, FileStatus), architectural rule documents (RuleViolation, Invariant, the serializable rule document interface), and version management (SchemaVersion, version constants, config schema validation guard).

The directory is organized into seven files, each focusing on a distinct responsibility:

- **`config.ts`** – Declares `SpineConfig`, the system's root configuration object, specifying schema version, project metadata, locale support, and optional LLM integration settings.
- **`index-documents.ts`** – The core data model: defines most interfaces used across the system, including `SpineUnit` (the top-level mirror unit), identity, semantic, skeleton, graph, provenance, rule violation, change intent, public surface, folder, project, and numerous union types (SourceLanguage, FileKind, SymbolKind, etc.).
- **`languages.ts`** – Contracts for language detection and support: `LanguageSnapshot`, `LanguageSupport`, `LanguageDelta`.
- **`manifest.ts`** – DTOs for the synchronization and manifest system: `SpineManifest`, `SyncBlock`, `DocRef`, `FileStatus`.
- **`rules.ts`** – Canonical structure for a rule document: fields like `schemaVersion`, `ruleId`, `title`, `appliesTo`, `severity`, `enforceable`, `bodyMarkdown`.
- **`versions.ts`** – Centralized version definitions: `SchemaVersion` type, current package/schema/config schema version constants, and a type guard for config schema compatibility.
- **`index.ts`** – Public barrel export that aggregates and re‑exports all modules above, providing a single import point for consumers of the types directory.

The most consequential implementation areas are the core data model (`index-documents.ts`), which defines the shape of every mirror unit and its substructures, and the configuration contract (`config.ts`), which governs project‑level setup and LLM integration. Version definitions (`versions.ts`) ensure cross‑module schema compatibility, while the manifest and rule modules maintain standardized DTOs for synchronization and architectural enforcement.
