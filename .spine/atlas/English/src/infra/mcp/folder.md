The `mcp` directory is the Model Context Protocol integration layer for ArchSpine, exposing the system’s internal resources and tools to external AI agents via the stdio transport. It acts as a gateway that controls context flow based on priming state and operational mode, ensuring agents receive only the information they are authorized to access.

The directory contains four infrastructure facade submodules, each with a well-defined responsibility:

- **context.ts** – Gating and context provisioning. Defines `MCPContextMode` (off, project-first, search-first), tracks internal boolean state for project and search priming, and uses `SEARCH_PRIMING_TOOLS` to update search priming when search-oriented tools are called.
- **resources.ts** – Resource templates and handlers. Provides URI templates (`spine://project`, `spine://file`, `spine://index`), reads and formats `.spine` directory files (JSON, Markdown), and enforces context‑gate access control via `MCPContextGate` (requireResourceAccess, noteResourceRead).
- **server.ts** – MCP server lifecycle. Initialises an MCP SDK Server with stdio transport, registers resource and tool request handlers, delegates to `SpineResources` and `SpineTools`, loads project context from Manifest and Config, and wraps errors via `toArchSpineError`.
- **tools.ts** – Tool definitions and execution. Wraps core ArchSpine components (Scanner, RuleEngine, Manifest, Config) into MCP tools such as `queryInvariants`, `queryResponsibilities`, `previewScan`, `getDriftHistory`, `getFileContext`, `getViewData`, `getViolationsSummary`, `listResourceTemplates`. It also provides baseline/sync status and integrates audit/tracing through `MCPContextGate`.

The most important implementation areas are **context‑aware access control** (determining when and how context flows to agents) and **resource/tool exposure** (making project metadata, file contents, scanning, rules, and drift history available through structured protocols). Concrete submodules like `MCPContextGate`, `SpineResources`, and `SpineTools` are the primary building blocks an implementer should focus on.
