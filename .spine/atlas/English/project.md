# ArchSpine Project Summary

ArchSpine is a *semantic mirror system* designed to transform source code repositories into structured, indexable, and AI-accessible architectural knowledge bases. It automates documentation generation, enforces architectural rules, and enables intelligent codebase analysis through a standardized mirror protocol. The system is organized into several cooperating layers and modules, each with a clear responsibility.

## Major Modules and Their Interactions

### 1. CLI Layer (`src/cli`)
The entry point for user interaction. It parses commands (`init`, `check`, `fix`, `sync`, `view` etc.), manages argument parsing, and delegates to core services. The `cli/commands` directory contains 21 adapters, one per subcommand. The `cli/init` subsystem handles project bootstrapping.

### 2. AST Layer (`src/ast`)
Parses source code into structured symbols across multiple languages. Language-specific grammars are stored in `src/ast/rules` (7 language definitions). This layer feeds symbols to the core engines for aggregation and analysis.

### 3. Core Engines (`src/engines`)
14 engine modules that perform:
- **File scanning** – discovers and indexes files.
- **Aggregation** – groups related symbols and metadata.
- **Architectural rule enforcement** – validates code structure against `.spine` rules.
- **Dependency analysis** – maps module dependencies.
- **System diagnostics** – checks health and consistency.

### 4. Infrastructure Layer (`src/infra`)
32 sub‑modules providing foundational services:
- **Configuration management** (`src/infra/config`) – loads/validates settings.
- **Persistence** (`src/infra/db`) – SQLite database lifecycle, schema evolution, and repositories for core entities.
- **LLM abstraction** (`src/infra/llm`) – provider‐agnostic interface with retry logic. Includes provider implementations (`src/infra/llm/providers`).
- **Manifest management** (`src/infra/manifest`) – tracks file states and integrity.
- **MCP server** (`src/infra/mcp`) – exposes ArchSpine resources and tools to external AI agents via the Model Context Protocol.
- **Prompt engineering** (`src/infra/prompt` and `src/infra/prompt-context`) – assembles prompts for LLM summarization.
- **Credentials** (`src/infra/credentials`) – secure secret storage.

### 5. Services Layer (`src/services`)
Coordinated orchestration of runtime command pipelines:
- **Check** – validates architecture against rules.
- **Fix** – applies automatic corrections.
- **Sync** – synchronizes the mirror with the source repo.
- **View generation** – produces architectural visualizations and analysis reports (`src/services/view` with 11 modules).

### 6. Task Pipelines (`src/tasks`)
13 modules that define sequential/parallel stages: reconciliation, AST extraction, summarization, validation, fixing, and committing. These pipelines are the “workflows” that run under services.

### 7. Supporting Directories
- **`.github`** – CI/CD workflows (3 workflows), issue templates.
- **`.spine`** – root configuration and rule engine. The `.spine/rules` directory contains architectural rulebooks.
- **`examples/demo-project`** – a working demo with multi‑language atlas (English, Simplified Chinese, en‑US), index, runtime checkpoints, and a sample source tree.
- **`schemas`** – JSON schema definitions for validating all mirror metadata.
- **`scripts`** – build, test, release, and demo automation.
- **`tests`** – 68 test suites across unit, integration, and e2e levels.

### How They Work Together in Practice

1. A user runs `archspine init` (CLI)` to bootstrap a `.spine` mirror directory in a repo.
2. On each `sync` command, the **CLI** delegates to the **service layer**, which triggers a **task pipeline**.
3. The pipeline first scans files using **engine** scanners, then parses source code via the **AST layer** into symbols.
4. **Aggregation engines** group symbols by module and store them in the **SQLite database** (infrastructure).
5. Architectural rules from `.spine/rules` are enforced by the **rule engine** (part of engines). Violations are reported.
6. If LLM summarization is enabled, the **prompt infrastructure** builds context, the **LLM provider** generates summaries, and results are stored in the mirror.
7. A **view service** can generate diagrams or reports from the stored data.
8. The **MCP server** allows external AI agents to query the mirror interactively.
9. Throughout, **CI/CD workflows** (e.g., GitHub Actions) run check/sync steps to keep the mirror up‑to‑date.

The system enforces strict import boundaries and naming conventions via the `.spine` rule system, ensuring the mirror remains consistent and actionable.
