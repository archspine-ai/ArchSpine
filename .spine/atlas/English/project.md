# ArchSpine: Semantic Mirror System

ArchSpine is a semantic mirror system that creates and maintains a structured, AI-readable reflection of a software project's architecture. It enables automated analysis, governance, and documentation by orchestrating a multi-stage pipeline that scans source code, extracts structural metadata, generates semantic summaries via LLMs, enforces architectural rules, and persists all artifacts in a mirrored `.spine` directory.

## Core Modules and How They Work Together

### 1. CLI Layer (`src/cli`)
The user-facing interface provides commands for all operations: `check`, `fix`, `sync`, `publish`, and initialization. It includes command adapters (`src/cli/commands/`) and a bootstrapping subsystem (`src/cli/init/`). The CLI delegates to the service orchestration layer.

### 2. Scanning and AST Parsing (`src/ast`)
The AST layer discovers source files and parses them into structured symbols using language-specific grammar definitions (`src/ast/rules/`). It supports multiple languages and extracts imports, exports, and type information. This raw structural data feeds into the core pipeline.

### 3. Core Pipeline Orchestration (`src/core`)
The core layer contains L2 aggregation logic and pipeline orchestration. It coordinates the flow from scanning through semantic generation to rule enforcement. It manages compatibility (`src/core/compat/`) and ensures data consistency across stages.

### 4. LLM Infrastructure (`src/infra/llm`)
This module provides client abstractions, configuration management, and resilient execution for semantic generation. It includes provider implementations (`src/infra/llm/providers/`) and retry mechanisms. The LLM layer transforms raw AST data into human-readable and AI-readable summaries.

### 5. Database and Persistence (`src/infra/db`)
A SQLite-based persistence layer manages indexing, state persistence, and data access. It includes repositories (`src/infra/db/repositories/`) for all core entities. The database stores metadata, manifests, and checkpoint records.

### 6. Service Orchestration (`src/services`)
The service layer coordinates the core pipelines: `check` (validate architecture), `fix` (apply corrections), `sync` (update mirror), and `publish` (generate documentation). It also manages runtime sessions and includes a view layer (`src/services/view/`) for generating architectural visualizations and analysis reports.

### 7. Rule Engine (`src/engines`)
The rule engine enforces architectural rules including layer isolation, naming conventions, and coding standards. It uses rule definitions stored in `.spine/rules/` directories. The engine integrates with the check and fix pipelines to validate and correct violations.

### 8. Task Pipeline (`src/tasks`)
Pipeline task implementations handle the core processing stages: file discovery, AST parsing, semantic generation, rule enforcement, and artifact persistence. Each task is a discrete step in the multi-stage pipeline.

### 9. Configuration and Credentials (`src/infra/config`, `src/infra/credentials`)
Configuration management handles system settings, while credential storage provides secure secret management for LLM API keys and other sensitive data.

### 10. MCP Integration (`src/infra/mcp`)
The Model Context Protocol layer bridges ArchSpine's internal capabilities with external AI agents, enabling automated analysis and governance workflows.

### 11. Templates and Prompts (`src/assets/templates`)
Template definitions for atlas documentation, prompts, rules, and views provide the structure for generated artifacts. The prompt infrastructure (`src/infra/prompt/`) constructs and assembles prompts for LLM interactions.

### 12. Mirror Directory Structure (`.spine`)
The `.spine` directory mirrors the project structure and contains:
- **Atlas**: Multi-language architectural documentation (English, Simplified Chinese, en-US)
- **Index**: Metadata and indexing records for source code
- **Rules**: Architectural governance specifications
- **Runtime**: Execution checkpoint records for synchronization runs

### 13. Schema Definitions (`schemas/`)
JSON schema definitions validate all metadata units, manifests, and rule documents, ensuring consistency across the system.

### 14. Test Suite (`tests/`)
A comprehensive test suite covers unit, integration, and end-to-end tests for all core components, including engine tests, infrastructure tests, and helper scripts.

## Practical Workflow

1. **Initialization**: The CLI bootstraps a project, creating the `.spine` mirror directory with default rules and configuration.
2. **Scanning**: The AST parser discovers source files and extracts structural metadata.
3. **Semantic Generation**: The LLM infrastructure transforms raw AST data into structured summaries.
4. **Rule Enforcement**: The rule engine validates the architecture against defined rules (layer isolation, naming conventions).
5. **Synchronization**: The sync pipeline updates the mirror with new artifacts and checkpoint records.
6. **Publication**: The publish pipeline generates documentation and visualizations.
7. **Continuous Integration**: CI/CD workflows (`.github/workflows/`) automate these steps on every commit.

The system is designed for both human consumption (Markdown documentation) and AI agent consumption (structured JSON metadata), making it a comprehensive solution for architectural governance and automated analysis.
